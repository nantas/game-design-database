# AGENTS.md

本文件定义 `game-design-patterns` 仓库内的工作约束，适用于整个仓库。

## 仓库目标

- 本仓库是一个可被 Obsidian 直接打开的游戏设计知识库。
- Markdown 页面是知识沉淀的唯一事实源。
- `src/` 与 `tools/` 中的代码只负责导入、抽取、渲染与验证，不维护独立数据库。

## 语言与输出

- 与用户沟通使用中文。
- 新建或修改的 Markdown 文档内容默认使用中文。
- 代码、测试名、模块名、命令名保持现有英文风格。

## 目录职责

- `10_来源/`
  - 记录来源站点或平台级说明。
  - 内容包括来源定位、可信度判断、常见内容类型、导入注意事项、相关入口页。

- `20_入口页/`
  - 记录分类页、导航页、`awesome-xxx` 汇总页等入口型页面。
  - 入口页只维护摘要、候选链接、后续处理建议，不直接承担模式总结。

- `30_网页卡/`
  - 记录单篇文章、README、文档页的沉淀结果。
  - 页面必须保留原始 URL、来源、导入时间、摘要、证据与关联模式。

- `40_设计模式/`
  - 记录可长期复用的机制、内容设计模式、设计策略与案例模式。
  - 这是 Agent 的主检索层。

- `50_专题索引/`
  - 记录人工维护的主题导航页。
  - 只做聚合和路由，不替代模式卡与网页卡。

- `90_模板与规范/`
  - 记录模板、命名规范、仓库说明等规则型文档。
  - 修改该目录前应确认是否会影响已有卡片结构。

- `src/game_design_patterns/`
  - 存放页面类型识别、HTML 抽取、Markdown 渲染、导入流程等核心实现。
  - 修改行为时必须优先补测试。

- `tools/`
  - 存放面向人工或 Agent 的命令入口。
  - 入口脚本应尽量保持薄封装，把真实逻辑放回 `src/`。

- `tests/`
  - 存放单元测试与 HTML fixture。
  - fixture 尽量使用真实页面快照，避免引入和实际站点结构不一致的人造样例。

## 数据入库规则

### 基本原则

- 单次导入的输入单位是一个 URL。
- 先判断页面类型，再决定落库位置。
- 查询默认依赖 Obsidian 搜索、`obsidian-cli` 与文本搜索，不额外开发查询数据库。

### 页面类型路由

- 文章页
  - 写入 `30_网页卡/`
  - 至少提炼一个模式候选
  - 若模式已存在，优先更新已有模式卡，而不是重复建卡

- 入口页
  - 写入 `20_入口页/`
  - 提取候选链接列表
  - 不做无限递归抓取

- 来源页
  - 写入 `10_来源/`
  - 只在引入新站点或需要补充站点规则时创建或更新

### 模式卡规则

- 模式卡采用“主词条 + aliases”规则去重。
- 文件名使用主词条。
- 同义词、英文名、社区叫法进入 `aliases`。
- 一个模式若只是新证据，不应新建第二张卡。

### 证据规则

- 模式卡中的关键结论必须能回链到网页卡。
- 网页卡必须包含原始 URL。
- 若页面可提取发布时间，则应保留。
- 摘要与证据优先保留正文信息，不要把作者简介、导航文案、站点噪音当作主要证据。

## Markdown 规则

- Frontmatter 必须从第 1 行开始，使用 `---` 包裹。
- 不允许重复 key。
- 多值字段使用 YAML 数组语法，例如 `tags: [game-design, pattern]`。
- 页面应尽量遵循模板中的固定二级标题，保证可检索性稳定。
- 网页卡必须链接到模式卡；模式卡必须回链网页卡。

## 开发规则

- 新功能、行为变更、抽取修复默认采用 TDD：
  - 先写失败测试
  - 确认失败原因正确
  - 再写最小实现
- 优先修根因，不做只掩盖症状的表面补丁。
- 不顺手修无关问题。
- 若修改抽取逻辑，优先补或更新 fixture 相关测试。
- 若修改 Markdown 结构，需确认不破坏现有模板与已生成样例。

## 验证规则

- 在声明任务完成前，必须运行与改动对应的验证命令。
- 常用验证命令：
  - `uv run pytest tests -v`
  - `uv run python tools/import_url.py '<url>'`
  - `rg -n -e '^type:|^aliases:|^## ' 10_来源 20_入口页 30_网页卡 40_设计模式`

- 涉及抽取修复时：
  - 先跑最小测试文件或单测用例
  - 再跑相关测试集
  - 最后再考虑全量测试

## Git 规则

- 每个完整任务使用一个清晰、单一主题的提交。
- 提交前保持工作区可解释，不混入无关变更。
- 提交信息使用现有风格，例如：
  - `feat: ...`
  - `fix: ...`
  - `docs: ...`
  - `test: ...`

## repo-agent / codex-agent 通用工作流

- 在派发任务前，先阅读本文件与相关模板、README、现有样例。
- 任务描述至少应包含：
  - 目标
  - 影响目录或文件
  - 输入 URL 或数据来源
  - 完成定义
  - 必跑验证命令
  - 是否需要提交与推送

- 推荐执行顺序：
  1. 检索现有条目，避免重复创建
  2. 确认改动边界
  3. 先写或补测试
  4. 做最小实现
  5. 运行验证
  6. 提交结果

- 若使用 `repo-agent` 或 `codex-agent` 驱动该仓库任务，任务提示中应显式引用本文件，并要求返回：
  - 修改文件列表
  - 验证命令与结果
  - 最终提交 hash

## 非目标

- 当前阶段不支持视频、社媒、登录墙页面、重度依赖 JavaScript 的正文抽取。
- 当前阶段不建立独立数据库或服务端检索层。
- 当前阶段不自动递归抓取整个站点。
