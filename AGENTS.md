# AGENTS.md

本文件定义 `game-design-patterns` 仓库内的工作约束，适用于整个仓库。

## 仓库目标

- 本仓库是一个可被 Obsidian 直接打开的游戏设计知识库。
- Markdown 页面是知识沉淀的唯一事实源。
- `src/` 与 `tools/` 中的代码只负责导入、抽取、渲染与验证，不维护独立数据库。
- 当前主线是"单游戏深入分析优先"，主实体是游戏主卡，不是来源页、入口页或网页卡池。

## 语言与输出

- 与用户沟通使用中文。
- 新建或修改的 Markdown 文档内容默认使用中文。
- 代码、测试名、模块名、命令名保持现有英文风格。

## 目录职责

- `10_游戏主卡/`
  - 仓库主检索层。
  - 每个游戏一个目录，固定包含五页：`入口页`、`核心体验`、`设计模式`、`游戏内容`、`证据索引`。

- `80_原则/`
  - 存放可复用的设计方法论与拆解原则。
  - 原则文档采用"短主文档 + 按需读取的细节页"结构。
  - 主文档只承载摘要、最小执行流程、继续阅读入口、参考游戏卡与校准状态。
  - 细节页必须明确说明"什么时候需要继续读这页"。
  - 当前原则：[[80_原则/核心体验拆解原则]]。

- `50_专题索引/`
  - 记录跨游戏或单游戏的人工专题导航。
  - 只做聚合和路由，不替代主卡固定五页。

- `90_模板与规范/`
  - 记录模板、结构规范、命名规范和仓库说明。
  - 修改该目录前应确认是否会影响已有主卡结构。

- `src/game_design_patterns/`
  - 存放 HTML 抽取、Markdown 渲染、导入流程等核心实现。
  - 修改行为时必须优先补测试。

- `tools/`
  - 存放面向人工或 Agent 的命令入口。
  - 入口脚本应尽量保持薄封装，把真实逻辑放回 `src/`。

- `tests/`
  - 存放单元测试与 HTML fixture。
  - fixture 尽量使用真实页面快照，避免引入和实际站点结构不一致的人造样例。

## 数据入库规则

### 基本原则

- 单次导入的输入单位是一个 URL，且必须绑定一个游戏名。
- 页面的第一身份是"该游戏的输入材料"，而不是独立沉淀中心。
- 默认不自动扩结构，不自动批量递归抓取。
- 当输入仅包含聚合页或目录页时，只能产出结构层判断，并在页面中显式标记证据缺口。

### 固定路由

- 新输入先登记到该游戏的 `证据索引`。
- `核心体验` 页面按 [[80_原则/核心体验拆解原则]] 方法论撰写：
  - 从宣传可感知、首局可体会的核心卖点起笔
  - 按真实决策-反馈循环逐层展开，不套固定层数模板
  - 循环层边界由新的决策目标、资源视角、重置点/回收点、策略影响共同定义
  - 高峰体验与首局体验连续但不混写
  - 玩家类型满足路径单列，不与循环层次混写
- 若材料已上升为可迁移设计判断，进入 `设计模式`。
- 若材料重点是内容池组织、分层和扩展，进入 `游戏内容`。
- `入口页` 只用于导航，不承载长正文分析。

### 扩结构规则

- 默认不扩结构。
- 仅当固定五页无法清晰承载主体观点时，才可提出扩结构建议。
- 提出扩结构前，必须先与用户确认。
- 首个样例验证通过前，不预建第六页或专题页体系。

## 证据规则

### 证据分层

按来源性质分层使用，不混用强度：

- **宣传层**：商店页、预告片文案、官方宣传材料
  - 用途：识别第一印象卖点、体验承诺
- **结构层**：wiki、规则文档、系统目录
  - 用途：确认系统存在、模块边界、可见规则
  - 边界：不能单独推出高确定性体验结论
- **体验层**：试玩报告、评测、玩家反馈
  - 用途：支撑玩家实际感受、决策价值、反馈强度
- **意图层**：开发者访谈、设计文档、GDC 演讲
  - 用途：支撑设计意图、目标玩家定位

### 结论强度分类

- **可确认判断**：被体验层或意图层证据稳定支撑
- **低风险判断**：结构层证据支撑，逻辑推理合理，但缺少直接体验验证
- **待验证判断**：当前证据不足，需要补录来源

### 证据索引要求

- 关键结论必须能回链到对应游戏的 `证据索引`。
- `证据索引` 必须保留原始 URL。
- 若页面可提取发布时间，应保留。
- 摘要与证据优先保留正文信息，不要把作者简介、导航文案、站点噪音当作主要证据。
- 显式标记证据缺口与待补来源方向。

## Markdown 规则

- Frontmatter 必须从第 1 行开始，使用 `---` 包裹。
- 不允许重复 key。
- 多值字段使用 YAML 数组语法，例如 `tags: [game-design, game-master-card]`。
- 页面应尽量遵循模板中的固定二级标题，保证可检索性稳定。
- `核心体验`、`设计模式`、`游戏内容` 的关键判断必须能在 `证据索引` 找到映射。
- 细节页必须包含 `parent` 字段回链主文档。

## 开发规则

- 新功能、行为变更、抽取修复默认采用 TDD：
  - 先写失败测试
  - 确认失败原因正确
  - 再写最小实现
- 优先修根因，不做只掩盖症状的表面补丁。
- 不顺手修无关问题。
- 若修改抽取逻辑，优先补或更新 fixture 相关测试。
- 若修改 Markdown 结构，需确认不破坏现有模板与已生成样例。

## 验证规则

- 在声明任务完成前，必须运行与改动对应的验证命令。
- 常用验证命令：
  - `uv run pytest tests -v`
  - `uv run python tools/import_url.py '<url>' --game '<game-name>'`
  - `rg -n -e '^type:|^tags:|^## ' 10_游戏主卡 50_专题索引 80_原则 90_模板与规范`

- 涉及抽取修复时：
  - 先跑最小测试文件或单测用例
  - 再跑相关测试集
  - 最后再考虑全量测试

## Git 规则

- 每个完整任务使用一个清晰、单一主题的提交。
- 提交前保持工作区可解释，不混入无关变更。
- 版本控制操作默认使用标准 Git CLI，不要求使用 GitButler 或 `but`，也不要将其作为任务前置条件。
- 提交信息使用现有风格，例如：
  - `feat: ...`
  - `fix: ...`
  - `docs: ...`
  - `test: ...`

## 工作流

### 核心体验页面撰写流程

1. 阅读入口：[[80_原则/核心体验拆解原则]]
2. 快速开始：当你只需要快速开始拆一个新游戏时，先读 `01_快速执行.md`
3. 按需深入：
   - 不确定某一层循环是否应该单独成层 → `02_循环层级判定.md`
   - 证据明确提示存在玩家分层问题 → `03_玩家类型补充.md`
   - 不确定证据够不够、或该用什么来源 → `04_证据分层与写作边界.md`
   - 写完一稿，想排查典型错误 → `05_常见误判.md`

### repo-agent / codex-agent 通用工作流

- 在派发任务前，先阅读本文件与相关模板、README、现有样例。
- 任务描述至少应包含：
  - 目标
  - 影响目录或文件
  - 输入 URL 或数据来源
  - 完成定义
  - 必跑验证命令
  - 是否需要提交与推送

- 推荐执行顺序：
  1. 检索现有游戏主卡，避免重复创建
  2. 确认改动边界
  3. 先写或补测试
  4. 做最小实现
  5. 运行验证
  6. 提交结果

- 若使用 `repo-agent` 或 `codex-agent` 驱动该仓库任务，任务提示中应显式引用本文件，并要求返回：
  - 修改文件列表
  - 验证命令与结果
  - 最终提交 hash
- 若任务包含提交、推送或分支操作，按仓库现有 Git remote / branch 工作流执行，不要求 GitButler 环境。

## 非目标

- 当前阶段不支持视频、社媒、登录墙页面、重度依赖 JavaScript 的正文抽取。
- 当前阶段不恢复入口页驱动的批量采集主线。
- 当前阶段不建立独立数据库或服务端检索层。
- 当前阶段不自动递归抓取整个站点。
- 玩家类型分层不是默认必写项，只有证据明确提示时才深入分析。
