# 设计模式拆解原则

## 这份原则解决什么问题
设计模式页用于沉淀“可复用的设计判断”，不是系统清单或内容目录。
本原则用于控制早期归纳噪声：只提炼支撑核心体验、且可在当前游戏内继续扩产的模式，不强行抽象所有内容条目。

## 最小执行流程
1. 从 `核心体验` 页提取体验锚点（高价值决策、反馈强度、学习链路）。
2. 在数据源中整理支撑这些锚点的系统/机制/内容条目。
3. 对每个条目判定是否满足“当前游戏内可扩产复刻”。
4. 满足条件者进入模式候选池；每轮最多讨论 3 条并由用户裁决。
5. 已确认模式回写 `设计模式` 页，并在 `游戏内容`、`证据索引` 建立回链。

## 模式准入规则
- 先看是否直接支撑核心体验；不支撑则不入模式。
- 再看是否可复刻：当前阶段以“当前游戏内可扩产”作为通过线。
- 不满足准入条件的条目，保留为内容实例、独立判例或待验证线索，不强行升级。
- 反模式不进入 `设计模式` 页，转由其他页面承接。

## 模式层级关系
- 设计模式页允许同时存在 `母模式`、`并列模式` 与 `细化方向`，不要求所有条目都落成严格树状结构。
- 母模式用于稳定表达主体验结构，例如 `分层循环可学习性模式`。
- 并列模式用于表达围绕同一核心体验、但关注点不同的模式簇成员，例如 `异质资源轴`、`基础强度模板`、`特定语境下重定价`。
- 细化方向用于表达某个模式下更偏机制设计指导的延展，例如 `资源转换模型` 可作为 `异质资源轴模式` 下的优先细化方向。
- 只有当“同类条目能提供更细机制/内容设计指导”时，才继续向下细化；否则保持并列，不强行拆子模式。
- 若某条细化方向要正式升级为稳定子模式，默认仍需满足同一游戏内至少两个不同机制/内容条目共同指向；若只有一条强证据，需先与用户确认。

## 与游戏内容/证据索引的回链规则
- `设计模式` 页只保留已确认模式与待验证模式线索。
- `游戏内容` 页需标注条目归属：`模式归属` 或 `未归属`。
- `证据索引` 页需维护“模式回链映射”，并标注可确认/低风险/待验证强度。
- 关键模式结论必须能回链到具体证据条目，不使用无证据的高确定性语气。
- 当某个模式被判断为“模式簇”时，`游戏内容` 页应优先回链到最具体、最有指导价值的模式，而不是一律只回链母模式。

## 用户共创确认机制
- 前期模式归纳必须与用户共同进行，采用小批次（每轮 3 条）裁决。
- 用户裁决优先级高于自动归纳；有争议的候选默认降级为待验证。
- 原则文档只沉淀已经过用户确认的规则，不预写未来可能规则。
