# 核心体验拆解原则

## 原则定义

核心体验不是对游戏系统的罗列，也不是对完整设计深度的事后总结。

核心体验是游戏向潜在用户直接传递、并且应当在玩家第一局中就能实际感受到的核心卖点。它必须具备独特性、不可替代性，并能通过持续的决策与反馈，平滑地引导玩家从初见体验进入更深层的高峰体验。

拆解核心体验时，应先回答两个问题：

1. 这款游戏在宣传与首次上手阶段，最想让玩家立刻感受到什么？
2. 这种感受是如何通过一层层决策-反馈循环，被持续放大并最终形成长期可反复游玩的高峰体验的？

因此，核心体验不是静态描述，而是一条体验递进链：第一印象卖点 -> 更小尺度的决策与反馈 -> 更大尺度的策略规划 -> 高峰体验 -> 不同玩家类型的满足路径。

## 为什么这样拆

如果只按系统模块拆解，最终写出来的往往是“这款游戏有什么”，而不是“这款游戏为什么让人想继续玩”。

系统、内容、机制都只是承载体验的手段。真正需要被记录的，是玩家如何在游玩过程中不断做出高价值判断，并从结果中获得明确、强烈、可学习的反馈。只有这种可学习的决策-反馈循环，才能把第一局的兴趣转化为长期游玩的动力。

优秀游戏的核心体验通常不是一次性给出的，而是分层展开的：

- 在第一印象阶段，它先让玩家迅速感知到一个鲜明卖点。
- 在初次上手阶段，它通过短周期决策和直接反馈建立“我能理解、我能进步”的感觉。
- 在熟练后，它继续把这些局部判断放大为更长周期的规划与构筑目标。
- 在高峰阶段，它让玩家感受到系统被自己驾驭后的强控制感、强收益感或强表达感。
- 对不同类型玩家，它还会提供不同门槛下的满足方式，而不是只服务单一理想玩家。

因此，核心体验拆解的目的，不是总结玩法说明书，而是识别一条从“宣传卖点”到“长期高峰体验”的连续路径，并说明这条路径是如何被设计出来的。

## 拆解步骤

拆解核心体验时，应按“从玩家最先感知到的体验入口，逐步推进到更大尺度的决策-反馈循环”来进行，而不是直接从系统结构或内容分类入手。

### 第一步：先确认第一印象卖点

先回答：游戏在预告片、商店页、宣传文案或开局前几分钟里，最希望玩家立刻感受到什么？

这一层只记录玩家能快速感知的体验承诺，例如：

- 节奏是紧张还是松弛
- 决策是高频还是低频
- 反馈是爆发式还是渐进式
- 玩家是在表达操作、表达策略，还是表达创造力

如果一个判断无法在宣传表达或首局前段中被玩家直接感知，它通常不应被写成核心体验的起点。

### 第二步：识别最小决策-反馈循环

找到玩家最频繁、最稳定、最先学会的那一层循环。

这里通常是“几秒到几十秒内完成一次判断并收到结果”的循环，例如：

- 一次出牌顺序
- 一次攻击与躲避交换
- 一次资源投入与即时收益判断
- 一次单位部署后的局部战场反馈

这一层要回答：

- 玩家在这一刻到底在权衡什么
- 正确与错误选择会带来什么明确差异
- 反馈是否足够强烈，足以让玩家形成学习

如果这一层没有清晰的高价值决策与强反馈，后续更大的体验层通常也站不稳。

### 第三步：沿真实游玩流程向上扩展循环层级

在最小循环之上，继续识别更大尺度的决策-反馈循环，直到覆盖该游戏的主要策略跨度。

这些层级不是固定模板，而应依据游戏本身来定。可能出现的层级包括：

- 单回合
- 单场战斗
- 单层或单章节推进
- 单局构筑或单局经营
- 多局继承或长期养成

每一层都要继续回答同样的问题：

- 这一层新增了什么决策权重
- 这一层的反馈是如何回流到下一轮判断的
- 它与更小层级之间是放大关系、筛选关系，还是纠错关系

只有当上下层循环之间存在连续传导，才能说明核心体验是被“层层放大”出来的，而不是彼此断裂的系统拼接。

### 第四步：识别高峰体验的形成条件

在覆盖主要循环层级之后，才进入高峰体验判断。

高峰体验不是简单的“变强了”，而是玩家在理解系统后，能够稳定感受到某种强烈且值得反复追求的状态，例如：

- 强控制感
- 强压制感
- 强连锁收益感
- 强表达感
- 强逆转感
- 强掌控复杂系统后的熟练满足感

这里要写清楚：

- 玩家需要先掌握哪些前置理解
- 哪些循环被打通后，高峰体验才会出现
- 高峰体验和第一印象之间是否存在连续路径，而不是断层跳跃

### 第五步：补充不同玩家类型的满足路径

核心体验不能只描述理想玩家的最终高峰，也要补充不同类型玩家如何被承接。

至少应检查：

- 轻度玩家是否能快速获得局部满足
- 普通玩家是否能稳定感到自己在进步
- 深度玩家是否能持续追求更高层的优化或表达
- 不同模式、难度或规则变体，是否在提供不同门槛下的体验兑现方式

如果某种满足路径只对熟练玩家成立，就不应被误写成“首局核心体验”；反过来，如果某种轻量爽感只能支撑短时体验，也不应被误写成游戏长期高峰体验。

### 第六步：最后再回写一句话总结

在完成以上拆解后，才能回写一句简洁的核心体验总结。

这句总结应同时满足三个条件：

- 能让未玩过的人理解这款游戏的独特卖点
- 能让玩过的人认出“对，就是这种感觉”
- 能被后续各层决策-反馈循环具体支撑，而不是空泛口号

## 层次判定规则

核心体验的层次不能预设为固定三层、四层或统一模板，而应根据该游戏真实存在的决策-反馈循环来判定。

判断层次时，不是看系统名称有多少层，而是看玩家是否在不同尺度上持续进行有区分度的判断，并收到能够影响后续行为的反馈。

### 1. 层次的基本判定标准

一个层次成立，至少应同时满足以下条件：

- 玩家在这一层有相对独立的决策对象
- 这一层的决策结果会在可感知时间内产生反馈
- 这一层的反馈会改变后续同层或更高层的判断
- 这一层不能被更小层完全替代，也不能只是更大层的空泛概括

如果某一层只是把下层重复描述一遍，或只是对上层做抽象命名，而没有新增决策权重与反馈意义，就不应单独成层。

### 2. 层次应按“从小到大”的真实循环展开

在第一印象之后，应优先从最小且最稳定的决策循环开始，再逐步扩大到更长周期的规划循环。

常见顺序可能包括：

- 单次操作
- 单回合
- 单场战斗
- 单层或单章节推进
- 单局构筑或经营
- 多局继承或长期养成

但并非所有游戏都包含这些层级，也并非所有层级都同等重要。应以该游戏真正驱动体验的循环为准，而不是机械套模板。

### 3. 判断“是否需要单独成层”的方法

判断某一层是否值得单独写出，不能只看它是否听起来更宏观，还要看这一层是否存在明确的资源边界、反馈闭环，以及一定程度上的“重置”或可回收资源。

可以用四个问题检查：

1. 这一层是否引入了新的决策目标？
2. 这一层是否有相对独立的资源约束？
3. 这一层结束时，是否存在明显的状态重置、资源回收或结算反馈？
4. 如果删掉这一层，核心体验的递进链是否会断裂或失真？

其中，第 3 点尤其重要。重置点或资源回收点，往往天然定义了玩家在这一层真正关心的决策范围，因此也是划分循环层次的高价值信号。

例如：

- 单回合结束后，防御值重置，说明这一层的核心判断更偏向“这一回合是否承受伤害、是否完成足够输出、是否值得为当前回合投入资源”
- 单场战斗结束后，战斗内消耗资源恢复，说明玩家可以把一些资源视为“本战斗内可全力使用”的临时资源
- 单层或单章节结束后，生命、补给、状态等若发生恢复或阶段性结算，说明这一层可能构成更大的资源规划循环

这些重置点不只是结构边界，也会直接反过来指导玩家策略：因为玩家知道某些资源在下一层级会恢复，所以会更激进地在当前层级内榨取收益；反之，如果资源无法回收，玩家就会更保守地规划使用。

因此，一个新的循环层次是否成立，通常取决于它是否同时带来了：

- 新的决策目标
- 新的资源视角
- 明确的阶段性反馈或重置
- 对玩家资源使用策略的实际影响

### 4. 上下层之间必须存在传导关系

层次不是并列罗列，而应形成连续传导。

需要明确写出：

- 下层的熟练度如何支撑上层规划
- 上层目标如何反过来约束下层操作
- 某层的失误会在哪一层暴露代价
- 某层的优化会在哪一层放大收益

如果上下层之间没有清晰传导关系，就说明拆出来的不是体验递进链，而只是若干系统说明。

### 5. 不同游戏允许存在不同的核心层数

有些游戏在最小循环和高峰体验之间只隔一层，有些游戏则会有更细的中间规划层。

例如：

- 强操作型游戏，可能从“瞬时判断”很快过渡到“整场节奏控制”
- 构筑或经营型游戏，可能需要经过“局部循环 -> 单局规划 -> 长期目标”多个层次
- 解谜或叙事游戏，某些层次可能不是资源规划，而是信息理解与预期修正

因此，层次数量不应作为质量标准。关键是是否完整覆盖了该游戏真正的核心递进路径。

### 6. 高峰体验不是普通层次的重复命名

高峰体验应视为层次链条的结果，而不是其中任意一层的别名。

只有当多个决策-反馈循环被玩家打通后，形成明显更强的控制感、收益感、表达感或掌控感时，才应进入高峰体验描述。

如果某种体验在第一局前几分钟就已完整兑现，它更可能属于第一印象或早期循环体验，而不是高峰体验。

### 7. 玩家类型不是循环层次的一部分

不同玩家类型的满足路径很重要，但它们不应和决策循环层次混写。

“轻度玩家获得什么满足”“高手玩家追求什么优化”，是对核心体验承接范围的补充说明，不是某个新的循环层级。

写作时应先完成：第一印象 -> 决策-反馈循环层级 -> 高峰体验

再补充：不同玩家类型如何在不同门槛上进入这条体验路径。

### 8. 证据不足时的处理原则

如果某一层在结构上似乎存在，但缺乏足够证据证明它对玩家体验真的成立，应标记为待验证，而不是先默认成层。

尤其要避免：

- 仅根据系统菜单推断层次
- 仅根据内容目录推断规划深度
- 仅根据高手玩法推断所有玩家都会经历该层次

## 玩家类型补充规则

核心体验不能只描述“理想玩家在完全理解系统后获得的最佳体验”，还应补充不同类型玩家如何以不同门槛进入这条体验路径，并分别获得可成立的满足。

如果只记录高水平玩家的最优体验，容易把“后期高峰”误写成“所有玩家都会感受到的核心体验”；反过来，如果只记录低门槛爽点，又会遗漏游戏真正支撑长期游玩的深层价值。

### 1. 玩家类型补充的目的

这一部分不是为了做完整用户画像，而是为了回答两个问题：

1. 哪类玩家最容易先感受到这款游戏的核心卖点？
2. 不同熟练度或偏好的玩家，分别会在什么门槛下获得什么满足？

其作用是防止核心体验被单一视角垄断，尤其是避免默认“高手体验 = 全部玩家体验”。

### 2. 玩家类型应围绕“体验兑现方式”来区分

在核心体验拆解中，不需要先做复杂人群细分，而应优先按“玩家通过什么方式获得满足”来划分。

常见区分角度包括：

- 更偏好快速正反馈，还是更偏好长期规划
- 更愿意接受高失败成本，还是更需要低压试错
- 更重视短时爽感，还是更重视系统掌控感
- 更容易被清晰目标吸引，还是更享受开放优化空间

因此，这里的玩家类型不是市场营销标签，而是体验承接路径的差异。

### 3. 先写主路径，再补替代路径

写作时应先明确：

- 游戏默认最想承接的主路径是什么
- 玩家通常通过哪些循环逐步进入高峰体验

然后再补充：

- 是否存在更低门槛的模式、规则变体或内容配置，让部分玩家可以更早获得局部满足
- 是否存在更高门槛的挑战结构，让深度玩家继续追求更强的优化与表达

也就是说，玩家类型补充是对主体验路径的展开，而不是脱离主路径另写一套体验。

### 4. 不同玩家类型的满足可以来自不同层级

不同玩家不一定在同一层级上获得满足。

例如：

- 有些玩家在最小循环里，只要感受到“我这一步打对了，反馈很明确”，就已经获得满足
- 有些玩家更在意单场战斗或单局规划中的资源掌控感
- 有些玩家只有在系统被完全打通、能够稳定达成高效循环时，才会感受到真正的高峰体验

因此，分析时应写清：

- 哪类玩家主要停留在哪个层级获得满足
- 哪类玩家会继续向更高层推进
- 游戏是否提供了足够顺滑的承接，让玩家愿意向上走，而不是在中途掉队

### 5. 模式、难度与规则变体是重要的承接工具

如果游戏存在不同模式、难度、角色起点或规则变体，应检查这些内容是否承担了“为不同玩家降低门槛或改变满足方式”的作用。

常见情况包括：

- 更宽松的模式让玩家更快感受到构筑成型或爽感兑现
- 更严格的模式要求玩家更早开始长期规划
- 特定角色或起始条件，让某类玩家更容易理解系统核心
- 特定规则变体，把复杂系统压缩为更短时间内可感知的结果

这些设计不只是内容差异，也是在主动分配不同玩家的进入路径。

### 6. 不要把“能玩”误写成“被满足”

玩家类型补充时，一个常见误判是：只要某类玩家可以完成流程，就默认他们已经获得了核心体验。

这是不成立的。

应区分：

- 玩家是否能继续玩下去
- 玩家是否真的感受到了这款游戏独特且值得重复体验的卖点
- 玩家获得的满足是短时补偿，还是已经进入核心体验主路径

如果某种模式只是降低挫败感，但没有让玩家感受到这款游戏真正的独特价值，就不能直接说它完整兑现了核心体验，只能说它改善了进入门槛。

### 7. 写作输出要求

在核心体验页中，玩家类型补充应尽量回答以下问题：

- 默认主路径更偏向哪类玩家
- 哪类玩家能最快获得第一层满足
- 哪类玩家会在更高层体验中获得持续动力
- 游戏提供了哪些机制、模式或规则来承接不同玩家
- 哪些满足是局部成立的，哪些满足才是完整高峰体验的一部分

这一部分的目标，不是覆盖所有玩家，而是说明核心体验的承接范围和分层兑现方式。

## 证据分层与来源优先级

拆解核心体验时，不同层次的判断应优先使用不同类型的数据源。不能用单一来源同时承担“宣传卖点”“系统结构”“玩家体验”“设计意图”四类判断。

### 1. 宣传层证据

用于判断游戏最希望潜在用户首先感知到的核心卖点，以及首局体验的入口承诺。

优先来源包括：

- 官方商店页文案
- 官方预告片标题、简介、字幕与可转写文本
- 官方宣传页
- 官方玩法介绍文章
- 媒体 preview / hands-on 首轮体验报道

这一层主要回答：

- 游戏被如何对外定义
- 哪些体验被突出强调为“立刻可感知的卖点”
- 哪些感受应当在首局或前几分钟被玩家直接体会到

如果某个判断无法在宣传层被感知，它通常不应作为核心体验的起点。

### 2. 结构层证据

用于判断游戏有哪些系统、资源、内容分层与可识别的循环结构。

优先来源包括：

- wiki 机制页与目录页
- 官方游戏手册、帮助页、系统说明
- 可验证的数据库型资料页
- 游戏内规则文本与界面信息

这一层主要回答：

- 玩家在不同尺度上可做哪些决策
- 游戏有哪些资源与状态变化
- 这些系统在流程上如何连接

结构层证据适合支撑“循环存在”“系统关系”“内容分层”等判断，但通常不足以单独支撑“体验强度”或“卖点独特性”的结论。

### 3. 体验层证据

用于判断玩家在实际游玩中感受到什么压力、反馈与学习路径。

优先来源包括：

- 实际游玩记录与可转写解说
- 深度评测与长文分析
- 高质量玩家复盘
- 多来源游玩观察交叉比对后的结论

这一层主要回答：

- 玩家在具体时刻到底在权衡什么
- 正确与错误选择是否会带来明确且强烈的后果差异
- 玩家是否能通过反馈快速形成学习
- 首局体验如何递进到更深层策略体验

如果没有体验层证据，就容易把“系统上存在”误写成“玩家确实强烈感受到”。

### 4. 意图层证据

用于判断开发者是否明确追求某种体验目标、学习曲线或玩家分层承接。

优先来源包括：

- 开发者访谈
- 开发日志
- 设计复盘
- 官方 FAQ、版本说明与模式设计说明

这一层主要回答：

- 开发者希望玩家学会什么
- 某个模式、难度或规则变体服务哪类玩家
- 某些体验路径是偶然形成，还是被明确设计出来的

意图层证据可以增强对“为什么这样设计”的判断，但不能替代实际玩家体验证据。

### 5. 来源组合原则

核心体验判断应尽量由多层证据交叉支撑，而不是依赖单一来源。

建议的组合方式是：

- 第一印象与核心卖点：以宣传层为主，必要时由意图层补强
- 决策循环层级：以结构层为主，必要时由体验层校正
- 反馈强度与学习路径：以体验层为主，不能只靠结构层推断
- 玩家类型满足路径：以意图层和体验层交叉判断
- 长期高峰体验：以体验层为主，必要时参考结构层解释形成条件

### 6. 当前仓库边界下的处理原则

在仓库工具尚未完整支持所有来源前，可以先按“理论上可使用的最优来源”定义方法，再根据工具能力选择可替代文本来源执行。

例如：

- 无法稳定处理视频本体时，可优先使用预告片简介、字幕转写、媒体 preview 或官方玩法文章作为宣传层替代
- 无法直接获得完整游玩记录时，可先以深度评测、复盘文章和结构层资料交叉形成低风险判断
- 证据不足时，应明确标记为“待验证”，而不是强行补齐

### 7. 常见来源误用

以下情况应避免：

- 用 wiki 聚合页直接推出高确定性的核心卖点
- 用系统说明直接替代玩家体验判断
- 用开发者自述直接替代玩家实际感受
- 用少量高手体验直接代表所有玩家类型
- 用长期高峰体验反推首局体验，而不检查中间递进是否成立

## 证据要求

核心体验的判断必须可回链到具体证据，但不同强度的判断，对证据的要求不同。

核心原则是：

- 先按判断类型选择合适来源
- 再按证据强度决定结论强度
- 证据不足时，明确标记边界，不补想当然的判断

核心体验不是纯主观感受记录，也不是只要有一句宣传文案就能成立。它必须建立在“宣传层 + 结构层 + 体验层 + 意图层”的合理分工之上，并最终回链到该游戏的 `证据索引`。

### 1. 每类判断应优先由对应证据支撑

不同判断应优先使用不同来源，不能混用或越级替代。

- 第一印象卖点
  优先使用宣传层证据，例如商店页文案、预告片文本、官方玩法介绍、媒体 preview
- 决策循环层级
  优先使用结构层证据，例如系统页、机制页、规则说明、界面信息
- 反馈强度与学习路径
  优先使用体验层证据，例如游玩观察、深度评测、复盘分析
- 模式设计与玩家分层承接
  优先使用意图层与体验层交叉验证，例如开发者访谈、模式说明、玩家实际体验对照

如果某个结论只被“错误层级”的证据支撑，就应降低确定性，或暂不下结论。

### 2. 结论强度必须匹配证据强度

可以把核心体验中的判断分为三类：

#### 2.1 可确认判断

满足以下任一条件时，可作为较高确定性的判断写入：

- 同一结论被两类以上证据交叉支撑
- 宣传层承诺与体验层观察一致
- 结构层可以解释其成立机制，体验层又能证明玩家确实感受到

这类结论可以直接写入核心体验页正文。

#### 2.2 低风险判断

当证据尚不足以完全确认，但已有较强线索，且结论仍然停留在较保守层级时，可以写为低风险判断。

常见情况包括：

- 宣传层明确强调，但体验层证据还不够完整
- 结构层能证明循环存在，但反馈强度仍缺实际体验验证
- 多个来源方向一致，但具体细节仍不稳定

这类内容可以写入正文，但应保留边界说明，避免绝对化表述。

#### 2.3 待验证判断

如果某个判断主要依赖推测、单一来源、高手经验泛化，或证据之间存在断层，应明确写为待验证，不应伪装成已成立结论。

常见情况包括：

- 仅根据 wiki 目录推断体验强度
- 仅根据开发者自述推断玩家一定会如此感受
- 仅根据高水平玩家玩法推断首局体验
- 仅根据少量材料推断不同玩家类型都能被满足

这类内容应进入“当前判断边界”或“待验证线索”，而不是直接进入核心判断。

### 3. `证据索引` 必须承担回链职责

核心体验页中的关键判断，必须能在该游戏的 `证据索引` 找到对应来源映射。

至少要保证：

- 关键卖点判断有原始来源链接
- 重要循环层次有对应结构或体验证据
- 高峰体验或玩家类型判断有明确证据支撑或边界声明
- 证据缺口被显式记录，而不是隐形跳过

如果某条结论写进核心体验页，却无法在证据索引中定位来源，这条结论就应被视为不合格。

### 4. 允许“判断先行”，但不允许“断言先行”

在真实分析中，研究者常常会先形成体验判断，再去找证据验证。这是正常的。

允许的流程是：

- 先提出一个工作假设
- 再根据判断类型补相应证据
- 证据不足时降低表述强度
- 最终只把被证据支撑的部分写成正式结论

不允许的流程是：

- 先凭印象写成确定结论
- 再用任意来源事后拼接解释
- 遇到证据不够时继续保留高确定性语气

也就是说，可以先有分析直觉，但正式沉淀时必须让证据来决定表述强弱。

### 5. 证据缺口本身也是有效输出

核心体验拆解不要求一次完成所有层次的高确定性判断。

如果当前只能确认：

- 第一印象卖点成立
- 存在若干明确循环层级
- 但高峰体验和玩家类型承接仍缺少体验层材料

那么正确做法是把已确认部分写清楚，并把缺口保留下来，而不是为了“完整”而强行补齐。

高质量的核心体验页，允许存在：

- 已确认判断
- 低风险判断
- 明确标注的待验证部分

这比写出一页表面完整、实际上证据失配的内容更有价值。

### 6. 常见证据错误

以下情况应视为证据使用不合格：

- 只用 wiki 聚合页推出核心卖点
- 只用系统说明推出玩家会感到“强反馈”
- 只用开发者话术证明某种体验已经兑现
- 只用高手玩法说明所有玩家都会进入高峰体验
- 没有证据索引回链，却在正文中下强结论
- 证据只能证明“存在这个系统”，却把它写成“这是核心体验来源”

### 7. 写作时的最低证据检查

在把某段内容写入核心体验页前，至少应快速检查：

1. 这条判断属于哪一类判断？
2. 它用了正确层级的证据吗？
3. 它的语气强度是否和证据强度匹配？
4. 它能否回链到证据索引？
5. 如果今天停止分析，这条判断是否仍然足够稳，不会误导后续 agent？

如果有任一问题答案是否定的，应降低表述强度，或移入待验证部分。

## 常见误判

核心体验最容易写错的地方，不是“写得不够多”，而是把不属于核心体验的内容误写成核心体验结论。以下误判应重点避免。

### 1. 把系统清单误写成核心体验

常见表现：

- 列出卡牌、遗物、地图、事件、角色等系统模块
- 然后直接把这些模块拼成一句“这款游戏的核心体验”

问题在于：

系统存在，并不等于玩家一定强烈感受到某种体验。系统只是体验的承载结构，不是体验本身。

应改为回答：

- 玩家在这些系统中到底在做什么高价值判断
- 哪些反馈让玩家明确感到“这一步对了/错了”
- 这些系统如何共同支撑一条连续的体验递进链

### 2. 把长期高峰体验误写成首局核心体验

常见表现：

- 直接把高熟练玩家的掌控感、极限构筑成型、复杂循环打通，写成游戏一上来就能兑现的体验
- 忽略玩家从初见到熟练之间的学习路径

问题在于：

核心体验要求玩家第一局就能开始感受到，但并不意味着第一局就能完整抵达最终高峰。首局体验和高峰体验必须连续，但不能混写。

应改为区分：

- 第一印象卖点是什么
- 玩家最早通过什么循环开始感受到价值
- 高峰体验需要哪些前置理解和能力积累

### 3. 把宣传口号直接当成体验事实

常见表现：

- 官方说“highly strategic”
- 就直接写“玩家会持续感到高价值决策”
- 官方说“every run is unique”
- 就直接写“每局差异都足够大且有意义”

问题在于：

宣传层证据只能证明“对外承诺了什么”，不能单独证明“玩家实际是否这样感受到”。

应改为：

- 用宣传层确定卖点入口
- 再用结构层与体验层验证这项承诺是否真的成立、成立到什么程度

### 4. 把结构可能性误写成体验强度

常见表现：

- 看到地图分支，就写“路线决策非常重要”
- 看到战斗日志，就写“玩家能高效复盘并形成策略学习”
- 看到构筑系统，就写“玩家会持续体验到深度优化乐趣”

问题在于：

系统提供了一种可能，不代表玩家一定会强烈感知到它，更不代表它一定是核心卖点。

应检查：

- 玩家是否真的频繁使用这一层判断
- 反馈是否足够直接和强烈
- 这一层是否真的影响后续策略，而不是边缘功能

### 5. 把高手体验误写成普遍玩家体验

常见表现：

- 用高手攻略、极限构筑、效率打法，总结整款游戏的核心体验
- 默认所有玩家都会经历同样的学习路径和高峰体验

问题在于：

高手看到的是系统被打通后的深层价值，但普通玩家、轻度玩家、首次上手玩家，未必会在同样门槛下获得相同体验。

应补充：

- 哪些满足是低门槛成立的
- 哪些体验只在熟练后才成立
- 游戏是否提供了不同模式或路径承接不同玩家

### 6. 把“能继续玩”误写成“已经获得核心满足”

常见表现：

- 玩家没有流失，就视为核心体验成立
- 某模式让玩家更容易通关，就视为核心卖点已经兑现

问题在于：

“留在流程里”不等于“感受到这款游戏独特且不可替代的价值”。

应区分：

- 流程可继续性
- 局部正反馈
- 核心体验的真正兑现
- 长期高峰体验的形成

### 7. 把层次拆成名字不同、实质重复的多层

常见表现：

- 写了很多层级名称
- 但每一层都只是“更大一点的策略”
- 没有新的资源边界、重置点、反馈方式或决策目标

问题在于：

这类拆法看起来完整，实际上只是抽象词堆叠，无法帮助后续 agent 理解游戏真正的体验结构。

应检查每一层是否都有：

- 新的决策目标
- 相对独立的资源视角
- 明确的结算、重置或反馈边界
- 对玩家策略使用的实际影响

### 8. 把内容丰富误写成体验丰富

常见表现：

- 角色多、卡多、敌人多、模式多
- 就直接写“体验层次丰富”或“长期可玩性强”

问题在于：

内容数量只能证明扩展面，不足以证明体验深度。真正决定体验丰富度的，是这些内容是否持续制造有意义的判断与可学习反馈。

应改为判断：

- 内容是否真的改变决策方式
- 内容差异是否会导向不同体验路径
- 内容扩展是否放大了原有核心体验，而不是只增加表面变化

### 9. 把设计意图误写成实际体验结果

常见表现：

- 开发者说“希望玩家感到更自由”
- 就直接写“玩家会感到更自由”
- 开发者说“这个模式是给轻度玩家的”
- 就直接写“轻度玩家一定会喜欢”

问题在于：

设计意图很重要，但它只能说明“想做什么”，不能单独证明“已经做到”。

应让意图层证据和体验层证据互相校正，而不是互相替代。

### 10. 为了完整而强行补齐缺失层

常见表现：

- 明明证据只够写第一印象和一两个循环层
- 仍然硬写高峰体验、玩家类型、长期结构
- 让整页看起来完整，但大量判断并不稳

问题在于：

这种写法会污染仓库，后续 agent 检索时会把不稳的猜测当作已验证知识。

正确做法是：

- 已确认的部分写清楚
- 低风险判断标明边界
- 待验证部分显式留下缺口

不完整但可靠，优于完整但失真。

## 参考游戏卡

原则卡应链接到已分析的具体游戏主卡，作为判断尺度、写法边界与类比方式的参考入口。

参考游戏卡的作用不是复制案例正文，而是回答：

- 这条原则目前主要是在哪些游戏上被观察到
- 哪些游戏能作为正向样例，说明原则如何成立
- 哪些游戏材料能帮助后续 agent 校准“什么能写、什么还不能写”

### 1. 链接原则

参考游戏卡应尽量链接到完整主卡或相关固定页，而不是零散来源。

优先链接对象包括：

- 游戏入口页
- 核心体验页
- 设计模式页
- 证据索引页

这样可以让后续 agent 同时看到：

- 当前结论是怎么写的
- 证据映射是否充分
- 哪些地方已经被修正，哪些仍有待验证

### 2. 参考方式

在原则卡中，参考游戏卡不应只列链接，还应简要说明“参考价值”。

推荐写法：

- `Slay the Spire 2`
  用于校准“从单回合决策到单局构筑的多层决策-反馈循环”如何支撑核心体验拆解；同时也用于提醒不要把 wiki 结构信息直接误写成体验结论。

### 3. 参考数量控制

原则卡起步阶段不应追求链接过多游戏。

更稳妥的做法是：

- 先保留少量高相关样例
- 明确这些样例主要校准哪一类判断
- 随着样例增加，再逐步扩展类比范围

样例数量少不是问题，关键是样例与原则之间的对应关系必须清楚。

### 4. 当前建议写法

- [[10_游戏主卡/Slay the Spire 2/Slay the Spire 2|Slay the Spire 2]]
  当前用于校准“核心体验应从第一印象卖点出发，按真实决策-反馈循环逐层展开到高峰体验与玩家类型满足路径”；同时作为“结构层证据不足以单独推出体验结论”的反例提醒。

## 当前校准状态

本原则当前处于早期校准阶段，主要用于指导首批游戏主卡的核心体验拆解，不应视为已经在所有类型游戏上充分验证的最终定稿。

### 1. 当前校准范围

当前主要基于以下观察形成：

- 单游戏主卡优先的仓库目标
- `Slay the Spire 2` 首样例中暴露出的典型问题
- 对“第一印象卖点 -> 决策-反馈循环 -> 高峰体验 -> 玩家类型满足”这一体验递进链的初步归纳

因此，这张原则卡当前最适合指导：

- 有明确决策反馈结构的游戏
- 可识别多层循环边界的游戏
- 需要区分首局体验与高峰体验的游戏

### 2. 当前已知有效点

目前已明确有效的部分包括：

- 核心体验不能等同于系统清单
- 核心体验应从宣传可感知、首局可体会的卖点起步
- 层次应按真实决策-反馈循环展开，而不是套固定模板
- 重置点、资源回收点和阶段性结算，是识别循环层次的重要信号
- 不同玩家类型的满足路径需要单独补充，不能默认高手体验代表全部玩家

### 3. 当前仍待继续校准的部分

以下内容仍需要更多游戏样例验证：

- 该原则对叙事驱动型、解谜型、强社交型游戏是否同样适用
- 当游戏缺少明显资源重置点时，循环层次应如何稳定识别
- 当宣传卖点与实际高峰体验差异较大时，应如何在核心体验页中分配权重
- 玩家类型补充规则在多人游戏或服务型游戏中的适配方式

### 4. 使用提醒

后续 agent 使用本原则时，应默认把它视为“当前仓库有效的方法草案”，而不是不可修改的最终规范。

如果新游戏样例明显冲击了本原则中的某一条，应优先记录冲突与边界，再决定是修订原则，还是为特殊类型游戏增加补充说明。
