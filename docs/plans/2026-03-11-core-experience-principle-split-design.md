# Core Experience Principle Split Design

## 背景

现有 [80_原则/核心体验拆解原则.md](80_原则/核心体验拆解原则.md) 已积累了完整的方法论，但正文达到数百行。对 agent 而言，这会带来两个直接问题：

- 首次装载上下文时成本过高，很多细节在当前任务并不需要。
- 方法、边界、误判、自检等内容混在同一页，agent 很难按任务阶段只读取必要部分。

本次目标不是改变原则内容本身，而是把它重组为“短主文档 + 明确按需读取的细节页”，让 agent 默认只装入最低必要信息，遇到具体问题时再顺着链接继续读取。

## 目标

- 保留 [80_原则/核心体验拆解原则.md](80_原则/核心体验拆解原则.md) 作为唯一入口主文档。
- 将长篇细节拆到 `80_原则/核心体验拆解原则/` 子目录。
- 主文档只承载原则摘要、最小执行流程、按场景继续阅读入口、参考游戏卡与当前校准状态。
- 每个子文档都要明确说明“什么时候需要继续读这页”，避免模糊索引。

## 非目标

- 不修改现有核心体验方法论的结论方向。
- 不扩展新的分析章节。
- 不同步修改其他原则文档结构。
- 不在本次中重写 `Slay the Spire 2` 或其他游戏主卡。

## 设计方案

### 方案 A：短主文档 + 主题细节页

采用一个短主文档作为总入口，把细节按主题拆成 5 页：

- `01_快速执行.md`
- `02_循环层级判定.md`
- `03_玩家类型补充.md`
- `04_证据分层与写作边界.md`
- `05_常见误判.md`

优点：

- 主文档最短，适合 agent 默认装载。
- 子页边界清楚，适合按任务阶段继续读取。
- 原有长文内容可以较稳定地分配到对应主题，不需要重写方法本身。

缺点：

- 主文档中的链接提示必须足够明确，否则 agent 仍可能继续读过多细节。

### 方案 B：短主文档 + 使用场景页

按“快速上手 / 证据不够 / 循环拆不清 / 玩家分层 / 自检”拆页。

优点：更贴近任务流程。

缺点：同一概念会散落在多个页面中，维护时更容易重复和漂移。

### 方案 C：保留长文，仅增加摘要区

在原主文档顶部加摘要和链接，正文下半部分保留。

优点：改动最小。

缺点：主文档依然过长，agent 很容易一次性装入整页，不能真正降低上下文成本。

## 采用方案

采用方案 A。

核心原因：需要真正降低默认上下文装载量，而不是只在长文顶部加一个摘要。主题拆页也最利于后续增量维护，因为“循环层级判定”“证据边界”“常见误判”这三类内容天然就是独立的按需参考模块。

## 文件结构

保留主文档：

- `80_原则/核心体验拆解原则.md`

新增子目录与细节页：

- `80_原则/核心体验拆解原则/01_快速执行.md`
- `80_原则/核心体验拆解原则/02_循环层级判定.md`
- `80_原则/核心体验拆解原则/03_玩家类型补充.md`
- `80_原则/核心体验拆解原则/04_证据分层与写作边界.md`
- `80_原则/核心体验拆解原则/05_常见误判.md`

## 主文档职责

主文档应压缩为短索引，只保留以下章节：

- `## 这份原则解决什么问题`
- `## 最小执行流程`
- `## 何时继续读取细节`
- `## 参考游戏卡`
- `## 当前校准状态`

主文档不再承载完整教学，不再承载长解释，不再展开所有边界条件。

## 细节页职责

### `01_快速执行.md`

用于“准备开始写某个新游戏的核心体验页，但暂时不需要深入处理边界争议”的场景。

承载内容：

- 核心定义
- 为什么这样拆
- 精简版拆解步骤
- 最低输出要求
- 何时继续读其他子页

### `02_循环层级判定.md`

用于“不确定某一层是否应单独成层”的场景。

承载内容：

- 循环层级判定原则
- 资源边界、重置点、回收点、阶段性结算
- 上下层传导关系
- 不应单独成层的情形

### `03_玩家类型补充.md`

用于“证据已明确提示玩家分层影响体验判断”的场景。

承载内容：

- 玩家分层不是默认必写项
- 只有当难度、节奏、策略深度、模式差异等证据明确出现分化时才深入分析
- 默认先写主体验路径，再按证据补充分层
- 不把玩家分层混入循环层级

### `04_证据分层与写作边界.md`

用于“不确定应该用什么来源、证据够不够、语气能写多强”的场景。

承载内容：

- 宣传层 / 结构层 / 体验层 / 意图层
- 可确认 / 低风险 / 待验证 的结论强度
- 最低写作检查
- 结构层来源不能单独推出高确定性体验结论的边界

### `05_常见误判.md`

用于“核心体验初稿完成后自检”的场景。

承载内容：

- 典型误判清单
- 最后自检问题
- 出现误判后应回读哪一页

## 链接设计原则

主文档中的链接不能只给标题，必须同时给触发条件。

推荐写法：

- 当你只需要快速开始拆一个新游戏时，先读 `01_快速执行.md`。
- 当你不确定某一层循环是否应该单独成层时，继续读 `02_循环层级判定.md`。
- 当证据明确提示存在玩家分层问题时，继续读 `03_玩家类型补充.md`。
- 当你不确定证据够不够、或该用什么来源时，继续读 `04_证据分层与写作边界.md`。
- 当你已经写完一稿，想排查典型错误时，继续读 `05_常见误判.md`。

关键要求：链接必须带有“什么情况下需要继续读”的说明，而不是只给一个裸链接。

## 内容迁移原则

- 不改变现有方法论结论，只重组位置。
- 主文档中的长段落一律迁出，只保留摘要和指路内容。
- 子页尽量避免重复解释；如需引用其他页，只放短提示并链接跳转。
- `03_玩家类型补充.md` 要额外加入一个边界：默认不要主动寻找所有玩家差异，只有证据明确提示时才深入，以避免噪音。

## 验证标准

完成后至少应满足：

- 主文档长度显著下降，能作为默认装载入口。
- 子页均带有“适用场景”或等价提示。
- 主文档的每个链接都带有明确读取条件。
- 所有新文档 frontmatter 合法，结构可被 `rg` 检查。
- 主文档与子页之间的链接可被 `rg` 检出。

## 风险与控制

### 风险 1：拆分后重复内容过多

控制方式：

- 主文档只保留摘要与索引，不保留长解释。
- 子页只保留本主题必须的信息，跨主题问题通过链接跳转。

### 风险 2：agent 仍然不知道该读哪一页

控制方式：

- 每条链接都写清触发条件。
- 每个子页开头写明“适用场景”。

### 风险 3：玩家分层继续被过度展开

控制方式：

- 在 `03_玩家类型补充.md` 中明确写入：玩家分层是证据触发的附加分析，不是默认主项。
