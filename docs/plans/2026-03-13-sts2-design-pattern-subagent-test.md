# StS2 设计模式拆解 Subagent 测试用例

## 当前迭代状态
- 当前阶段：第 4 轮原则-测试联调后，测试框架已稳定为“半盲测试”。
- 当前结论：原则文档的修改会直接影响 subagent 的拆解质量，测试框架可持续复用。
- 当前已稳定验证：
  - `设计问题层` 的命名能提升 subagent 对随机性相关模式的抽象层级。
  - `共享背景规则不等于共享设计问题` 能提升相邻模式边界质量。
  - `母模式关联优先 + 不清晰先降级` 能抑制附会式归类。
- 当前主要卡点：
  - subagent 仍会在“边界稍不清晰但设计问题已成立”的情况下过度保守。
  - 随机性控制类模式容易被压回待验证，而不是先以高层问题立项。
  - `生命周期管理` 与 `双层生存账本` 这类模式仍不够稳定，说明“何时先立项、何时先降级”的规则还不够清楚。
- 当前重点关注问题：
  - 如何明确区分：
    - “设计问题已经成立，但细化证据不足”
    - “设计问题本身就还没站稳”
  - 如何让 subagent 在不读取现成答案页的前提下，稳定识别：
    - 随机性控制相关模式
    - 生命周期管理模式
    - 双层生存相关模式
- 下轮优先改进方向：
  - 在原则中补“允许先在设计问题层立项，具体实现后补”的规则。
  - 继续用半盲测试验证该规则是否能减少过度保守。

## 测试目标
验证一个只被提供 `设计模式拆解原则` 与 `obsidian-cli` 检索能力的 subagent，是否能够基于仓库内 `Slay the Spire 2` 的数据源，独立完成一次符合设计师视角的设计模式拆解。

本测试不测“是否复现现有答案”，而测：
- 是否能够从设计问题出发提出模式，而不是复述系统名词。
- 是否能够做出准入、边界、强度、细化判断。
- 是否能够正确控制误判，不把外围系统或弱证据条目抬升为主模式。

## 测试范围
测试对象只针对 `Slay the Spire 2` 当前仓库内数据源。

当前判定基准以现有游戏卡和原则文档为准，不要求 subagent 立即跨游戏泛化。

## Subagent 输入约束
### 允许输入
- [[80_原则/设计模式拆解原则]]
- `obsidian-cli` 检索能力
- 仓库内通过检索召回到的 `Slay the Spire 2` 相关页面
  - 例如 `核心体验`
  - `证据索引`
  - 相关规则页、专题页、原则页

### 禁止输入
- 不允许直接读取现成答案页：
  - `10_游戏主卡/Slay the Spire 2/Slay the Spire 2 - 设计模式.md`
- 不允许读取带有现成模式归属结构的辅助页：
  - `10_游戏主卡/Slay the Spire 2/Slay the Spire 2 - 游戏内容.md`

### 原因
- 该限制用于确保测试的是“原则 + 检索”能否支持独立拆解，而不是复述现有模式页。

## 测试任务
subagent 需要完成以下任务：
1. 基于允许检索到的数据源，整理 `Slay the Spire 2` 的设计模式候选。
2. 对每个候选明确写出它回答的设计问题。
3. 对每个候选做以下判断：
   - 准入判断
   - 边界判断
   - 对核心体验的支撑强度
   - 是否继续细化
4. 明确哪些条目应降级为：
   - 内容实例
   - 待验证
   - 不纳入主模式

## 期望输出格式
subagent 的输出至少应包含以下部分：

### 1. 检索与输入范围
- 它读取了哪些页面
- 为什么这些页面与当前拆解相关

### 2. 模式候选清单
- 每个候选模式的名称
- 每个候选模式回答的设计问题

### 3. 判断过程
- 准入判断
- 边界判断
- 强度判断
- 细化判断

### 4. 最终结论
- 已确认模式
- 待验证模式
- 不纳入模式的条目与原因

## 核心评分维度
### 1. 模式质量
- 模式是否在回答设计问题，而不是复述系统或规则名词。
- 模式是否能反向指导后续设计扩产。

### 2. 边界质量
- 是否正确区分相邻模式。
- 是否避免把同一设计问题拆成多个近义模式。

### 3. 强度质量
- 是否区分直接支撑、底盘支撑、间接支撑。
- 是否避免把扩展型模式抬升为核心玩法同级。

### 4. 证据质量
- 是否基于可检索到的条目做判断。
- 是否避免循环引用和无证据高确定性语气。

### 5. 误判控制
- 是否错误纳入 `跨 Act` 决策为主模式。
- 是否错误纳入 `Event Log` 为已确认模式。
- 是否错误把预算、洗牌、手牌窗口混成同一模式。

## 通过线
满足以下条件可视为通过：
- 至少命中以下 5 个核心模式中的 4 个：
  - `分层强决策反馈循环`
  - `抽牌-能量双预算模式`
  - `生命-防御双层生存模式`
  - `洗牌循环控制模式`
  - `生命周期管理模式`
- 不得将 `跨 Act` 决策升级为主模式。
- 不得将 `Event Log` 升级为已确认模式。
- 必须至少对 3 个模式明确写出“它回答的设计问题是什么”。
- 必须至少出现一次边界判断，解释两个相邻模式为什么应拆开或合并。

## 重点观察项
本轮测试特别关注以下问题：
- subagent 是否能用“设计决策单元”而不是“百科分类单元”来拆模式。
- subagent 是否能区分：
  - `抽牌-能量双预算模式`
  - `洗牌循环控制模式`
  - `手牌窗口管理模式`
- subagent 是否会把资源扩展层模式错误写成与基础玩法层同等强度。

### 评分
- 模式质量：中

  - 问题：把 `异质资源轴`、`重定价`、`资源转换` 过早压成一个聚合模式。
- 边界质量：中偏弱
  - 优点：能区分“预算”和“威胁遥测”。
  - 问题：没有成功拆开 `洗牌循环控制` 与 `手牌窗口管理`，反而把三类问题重新聚合成“随机性可管理化”。
- 强度质量：中

  - 问题：把 `生命周期管理` 降成实现层，说明对“底盘模式”独立性的判断还不稳。
- 证据质量：中

  - 问题：读取了 `游戏内容` 页，而该页包含大量指向现成模式标题的锚点链接，导致测试并非完全黑盒。
- 误判控制：中偏强
  - 优点：没有把 `跨 Act` 或 `Event Log` 升成主模式。
  - 问题：未能识别 `生命周期管理模式` 为已确认模式，属于漏判而非误判。

### 结论
- 是否通过：未通过
- 未通过原因：
  - 未满足“至少命中 5 个核心模式中的 4 个”这条通过线。
  - 实际稳定命中：
    - `分层强决策反馈循环`
    - `抽牌-能量双预算模式`
    - `生命-防御双层生存模式`
  - 未稳定命中：
    - `洗牌循环控制模式`
    - `生命周期管理模式`
- 原则缺口：
  - 当前原则虽然强调“按设计问题拆解”，但对“底盘模式如何与实现层区分”的说明仍不够强，导致 subagent 容易把 `生命周期管理` 并回实现层。
  - 当前原则虽然举了 `预算 / 随机性干预 / 窗口宽度` 的样例，但还缺少一条更硬的约束：当三个候选分别回答不同问题时，不应因为共享 `Draw` 规则背景而重新聚合。
- 测试用例缺口：
  - 首轮运行允许读取 `游戏内容` 页，这会暴露现有模式名称和结构，降低黑盒纯度。
  - 因此从第二轮起，本测试升级为半盲测试，禁止读取 `游戏内容` 页，只允许读取 `核心体验`、`证据索引` 与专题/规则页。
- 下一轮改进建议：
  - 把测试升级为“半盲测试”：
    - 允许：`原则页`、`核心体验`、`证据索引`、专题页、规则页
    - 禁止：`设计模式`、`游戏内容`
  - 在原则页补一条更明确的边界规则：
    - “共享同一背景规则，不等于共享同一设计问题。”
  - 在原则页补一条底盘模式判定规则：
    - “如果移除该机制会让多个内容类型失去统一结算框架，它更可能是独立底盘模式，而不是实现细节。”
